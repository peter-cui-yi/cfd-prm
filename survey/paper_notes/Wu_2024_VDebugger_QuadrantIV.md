# VDebugger: Harnessing Execution Feedback for Debugging Visual Programs

## Basic Information

**Title**: VDebugger: Harnessing Execution Feedback for Debugging Visual Programs

**Authors**: Xueqing Wu, Zongyu Lin, Songyan Zhao, Te-Lin Wu, Pan Lu, Nanyun Peng, Kai-Wei Chang

**Venue**: Findings of the Association for Computational Linguistics: EMNLP 2024

**Year**: 2024

**Link**: https://arxiv.org/abs/2406.13444 | https://aclanthology.org/2024.findings-emnlp.575 | https://github.com/shirley-wu/vdebugger

## Abstract Summary

VDebugger is a critic-refiner framework that debugs visual programs—executable Python code generated by LLMs for visual reasoning—by tracking execution step-by-step and leveraging detailed execution feedback to localize and correct program logic errors. Training data is generated via a novel mask-best decoding technique that automatically injects bugs into correct programs. Evaluated on six datasets, VDebugger improves downstream accuracy by up to 3.2%, with 2.3% generalization on unseen tasks.

## Methodology Analysis

### Representation Type
- [x] Structured Trace (Python programs with subroutine calls to specialist vision models)

### Verification Channel
- [x] Execution Feedback (step-by-step program execution; runtime return values and exceptions fed back to critic-refiner)

### 2×2 Matrix Placement
**Quadrant**: IV

**Justification**:
- **Structured**: Reasoning traces are Python programs with explicit function signatures (`detect_objects()`, `count()`, `compare_attributes()`), not free-form text. Each step has a defined input/output contract that can be machine-checked.
- **Tool/Execution**: VDebugger tracks program execution step by step, capturing execution feedback (return values, intermediate results, exceptions) and using that to localize errors. The core insight is that *execution output* (not LLM introspection) drives debugging.
- **Replayability**: Complete program trace + execution log can be fully replayed. Bug location can be pinpointed to a specific code line. Fixed programs can be re-executed to verify the correction.

## Key Contributions

1. First critic-refiner framework specifically for visual program debugging, using execution feedback (not LLM self-reflection) for error localization
2. Mask-best decoding: novel automated technique for generating (buggy program, execution trace, corrected program) training triplets without human annotation
3. Generalization to unseen compositional tasks: 2.3% improvement on COVR (out-of-distribution), demonstrating learned debugging is transferable

## Verifiability Analysis

### Grounding Strength
Strong — each program step is grounded in a specific vision model API call (e.g., object detection, VQA). The grounding is explicit in the code itself, not inferred from text.

### Checkability
High — execution output at each step is a concrete value (list of objects, count, boolean, etc.) that can be programmatically checked against expected types and plausibility constraints.

### Replayability
High — the program is deterministic (given fixed API implementations), so any query's reasoning can be replayed exactly. The debugging trace itself records which execution steps produced correct vs. incorrect intermediate results.

### Faithfulness Risk
Low — unlike text CoT, the program is the reasoning: there is no post-hoc rationalization. The program either executes correctly or it doesn't; execution output is the ground truth, not model-generated explanation.

### Robustness
Moderate — failure in one step propagates downstream. If critic misidentifies the buggy step, the refiner fixes the wrong part. Performance degrades on highly compositional queries where multiple steps have subtle errors.

### Cost/Latency
Medium-High — multiple rounds of (execute → critique → refine → re-execute) increase computational cost. Each iteration requires re-running the full program plus a forward pass through critic and refiner models.

### Security
Medium — executing LLM-generated code requires sandboxing. Visual API calls may trigger unintended resource access. Masked decoding for training may occasionally produce programs with unintended side effects.

---

## Failure Modes

1. **Critic mislocalization**: If execution feedback is ambiguous (e.g., wrong return value at step 3 is caused by step 1), the critic may patch the wrong step, leading to no improvement or new bugs
2. **Cascading errors**: A single wrong intermediate result (e.g., wrong bounding box from step 2) propagates through all downstream steps; no mechanism to backtrack more than one step
3. **API-level failures**: Underlying specialist models (object detector, VQA model) may produce wrong outputs that look syntactically correct (no exception raised), making bugs invisible to the execution-based critic

## Evaluation

### Metrics Used
- [x] Answer Accuracy (downstream task accuracy on six datasets)
- [x] Step Correctness (implicit in debugging success rate)
- [x] Trace Replayability (programs are executable and re-runnable by design)

### Benchmarks
GQA, NExT-QA, STAR, TallyQA, WebQA, COVR

### Key Results
- Up to 3.2% downstream accuracy improvement across six datasets
- 2.3% improvement on unseen COVR task (zero-shot generalization)
- 58% of visual program errors are logic errors (not generation/format failures) — motivating the entire approach

## Training & Alignment

### Method
- [x] SFT with Rationale (trains critic and refiner on (buggy program, execution trace, fixed program) triplets)
- [x] Other: mask-best decoding for automated training data generation

### Data Collection
Automated pipeline: (1) generate correct programs on training data; (2) inject errors using mask-best decoding (mask a code token, sample alternatives, keep ones that produce wrong output); (3) run injected programs, capture execution trace; (4) use original correct program as supervision target.

## Quotes & Key Insights

> "58% of the total errors are caused by program logic errors." — This motivates the entire paper: most failures are NOT generation failures but logical mis-steps in otherwise syntactically valid code.

Key insight: **execution feedback as a debugging oracle** is more reliable than LLM self-critique because it provides *ground truth intermediate states* rather than model-generated assessments of correctness.
