## Abstract Summary

Artemis addresses the limitation of purely linguistic intermediate reasoning in visual perception tasks by introducing structured proposal-based reasoning where each intermediate step is represented as a (label, bounding-box) pair capturing a verifiable visual state. Built on Qwen2.5-VL-3B, the framework enables explicit tracking of intermediate states, direct supervision for proposal quality, and avoids ambiguity from language-based reasoning, achieving strong performance on grounding, detection, counting, and geometric-perception tasks.

---

## Methodology Analysis

### Representation Type
- [x] **Structured Trace** — (label, bounding-box) pairs representing visual states. Each reasoning step is a structured spatial token consisting of an object label and its spatial localization, not free-form natural language descriptions.

### Verification Channel
- [x] **No Tools / No Execution** — reasoning is self-contained within the model's forward passes. The (label, bbox) pairs are generated by the model itself without invoking external detectors, localization tools, or execution environments.

### 2×2 Matrix Placement
**Quadrant**: **III (Structured w/o Tools)**

**Justification**:
- **Structured**: Intermediate reasoning states are (label, bbox) pairs — highly structured spatial representations with clear semantics. Each token encodes both semantic (label) and spatial (bounding-box) information in a fixed format, fundamentally different from free-form text.
- **No Tools**: All reasoning occurs within the model. The bounding boxes are predicted directly by the VLM without calling external object detectors or localization APIs. The structured format is internal to the model's output space.

---

## Key Contributions

1. **Structured spatial token representation**: Proposes representing intermediate reasoning steps as (label, bounding-box) pairs instead of natural language, aligning reasoning representation with the spatial and object-centric nature of visual perception tasks.

2. **Explicit state tracking and supervision**: The structured format enables direct supervision for proposal quality (both label accuracy and localization precision) and explicit tracking of intermediate reasoning states throughout the perception pipeline.

3. **Perception-policy learning framework**: Integrates structured reasoning into reinforcement learning for perception policy, demonstrating that spatially grounded reasoning provides a principled route toward scalable and general perception policies.

4. **Strong generalization**: Built on Qwen2.5-VL-3B, achieves strong performance on grounding and detection tasks with substantial generalization to counting and geometric-perception tasks, plus competitive performance on general MLLM benchmarks.

## Verifiability Analysis

| Dimension | Rating | Reasoning |
|-----------|--------|-----------|
| **Grounding** | **High** | (label, bbox) pairs are directly grounded in visual content — labels correspond to detected objects and boxes provide spatial localization. This is stronger grounding than text-based reasoning. |
| **Checkability** | **High** | Bounding boxes can be automatically checked against ground truth using IoU metrics. Labels can be verified for correctness. The structured format enables precise, quantitative evaluation of intermediate steps. |
| **Replayability** | **High** | Given same model weights and input, same (label, bbox) pairs are produced (assuming deterministic decoding). The structured format is serializable and can be logged for later analysis. |
| **Faithfulness Risk** | **Medium** | While more grounded than text, the model might still generate plausible-looking (label, bbox) pairs that don't correspond to actual visual evidence. However, the spatial constraint makes hallucination harder than with free-form text. |
| **Robustness** | **Medium-High** | Structured representation provides robustness against linguistic ambiguity. However, performance may degrade on objects with unusual appearances or spatial configurations not seen during training. |
| **Cost/Latency** | **Medium** | Generating structured (label, bbox) pairs may have similar cost to text generation. The structured format might actually reduce reasoning chain length by avoiding verbose linguistic descriptions. |
| **Security** | **Low** | No external tool calls reduce attack surface. The structured output format constrains the output space, potentially reducing vulnerability to prompt injection attacks. |

**Q3 vs Q1**: Q1 methods express reasoning as free-form text tokens. Artemis's (label, bbox) pairs are structured spatial tokens — much more compact and directly verifiable against visual ground truth. Text reasoning might say "there's a cat in the upper left corner" while Artemis outputs ("cat", [x1, y1, x2, y2]) — the latter is precise and measurable.

**Q3 vs Q4**: Q4 methods produce executable artifacts (code, programs). Artemis's (label, bbox) pairs are not executable but are directly verifiable against visual ground truth using IoU calculations. The verification is simpler than code execution but still provides quantitative feedback.

## Failure Modes

1. **Localization errors**: Incorrect bounding box predictions (poor IoU) lead to downstream reasoning errors, even if the label is correct. The model might localize the wrong object instance.

2. **Label-localization mismatch**: The model might predict a correct label but associate it with the wrong spatial location, or vice versa — accurate box around the wrong object category.

3. **Cascading proposal errors**: Early errors in the reasoning chain (incorrect proposals) propagate through subsequent steps, with no mechanism for backtracking or correction.

4. **Limited expressiveness**: (label, bbox) pairs may be insufficient for complex reasoning requiring attributes, relationships, or multi-object interactions beyond simple detection.

5. **Ambiguity in crowded scenes**: For scenes with many similar objects, the model might struggle to distinguish and correctly localize individual instances, leading to duplicate or missing proposals.

6. **Scale sensitivity**: Performance may degrade for objects at unusual scales (very small or very large) not well-represented in training data.

### Data Collection
Training data includes images with bounding box annotations and associated labels. The model learns to generate (label, bbox) pairs as intermediate reasoning steps, with direct supervision on both label accuracy and localization precision (IoU). The perception policy is learned through reinforcement learning with rewards based on task success and proposal quality.

**Key Design**: Unlike language-based reasoning chains that perform semantic reasoning in unstructured linguistic space, Artemis performs reasoning in spatial and object-centric space — matching the representation to the task requirements.

## Quotes & Key Insights

> "Recent reinforcement-learning frameworks for visual perception policy have begun to incorporate intermediate reasoning chains expressed in natural language. Empirical observations indicate that such purely linguistic intermediate reasoning often reduces performance on perception tasks."

> "We argue that the core issue lies not in reasoning per se but in the form of reasoning: while these chains perform semantic reasoning in an unstructured linguistic space, visual perception requires reasoning in a spatial and object-centric space."

**Key Insight**: Artemis identifies a fundamental mismatch between reasoning representation (language) and task requirements (spatial perception). The insight that "visual perception requires spatial reasoning" motivates the shift from linguistic to spatial representations.

**Design Principle**: "Align reasoning representation with task space" — for perception tasks, use spatial representations; for language tasks, use linguistic representations. This principle could generalize to other domains (e.g., using temporal representations for video reasoning).

**Verification Advantage**: The (label, bbox) format enables direct quantitative evaluation using IoU — a significant advantage over text-based reasoning where correctness is harder to measure automatically.

### Narrative Role
Artemis serves as a representative example of **spatial token-based reasoning** in Quadrant III. It demonstrates how matching the reasoning representation to the task domain (spatial tokens for visual perception) improves performance and verifiability. The paper supports the survey's argument about the importance of representation design in Q3 methods and the trade-offs between different structured representation choices.

### Comparison Points
- **Excels at**: Direct visual grounding, quantitative verification (IoU), compact representation, strong empirical performance on perception tasks
- **Fails at**: Complex relational reasoning, attribute-based reasoning, handling ambiguous or crowded scenes
- **Contrast with G2**: Artemis uses simple (label, bbox) pairs vs G2's rich scene graphs — less expressive but more directly verifiable
- **Contrast with MCOUT**: Artemis uses human-interpretable spatial tokens vs MCOUT's latent vectors — fully interpretable and measurable
- **Contrast with COGT**: Artemis uses spatial structure vs COGT's linguistic structure — better suited for perception tasks

## Notes

- This paper represents the "spatial tokens" sub-category of Q3 structured representations, complementing G2's "scene graphs", COGT's "causal graphs", and MCOUT's "latent vectors".
- The (label, bbox) format is elegantly simple yet powerful — minimal structure needed for direct visual grounding and verification.
- The key insight about representation-task alignment ("visual perception requires spatial reasoning") is a valuable design principle for other domains.
- Strong generalization from grounding/detection to counting/geometric tasks suggests the structured representation captures fundamental visual reasoning capabilities.
- The perception policy learning framework could be extended to embodied AI and robotics applications where spatial reasoning is crucial.
- Should be cited alongside G2, COGT, and MCOUT as representative Q3 methods with different structured representation choices, each suited for different task types.
