## Abstract Summary

MM-Zero is the first zero-data self-evolving VLM framework. It employs a three-role architecture: (1) **Proposer** generates concepts and questions; (2) **Coder** converts concepts into executable Python/SVG code to render images; (3) **Solver** performs multimodal reasoning on the generated images. All roles are trained with GRPO, integrating execution feedback and visual verification. The framework enables VLMs to evolve from zero human-annotated data through self-generated synthetic training data.

## Methodology Analysis

### Representation Type
- [ ] Textual Rationale (free-form CoT, plans, reflections)
- [x] Structured Trace (Coder generates executable Python/SVG code; Solver produces reasoning traces; Proposer outputs structured concepts and questions)

### Verification Channel
- [ ] No Tools / No Execution
- [ ] Tool-Augmented (detectors, OCR, retrieval, code execution, web search)
- [x] Execution Feedback (Coder's code is executed to render images; execution success/failure and visual output provide reward signals)

### 2×2 Matrix Placement
**Quadrant:** IV (Structured Traces + Execution Feedback)

**Justification:**

1. **Coder Generates Executable Code**: The Coder role produces Python/SVG code that is executed to render images. This code has formal syntax, is runnable, and produces deterministic (or reproducible) visual outputs. The reasoning trace includes executable programs, not just text.

2. **Execution Feedback as Reward Signal**: The Coder's code is executed; the resulting images (or execution errors) provide direct feedback. Execution success, visual quality, and alignment with the Proposer's concepts feed into the GRPO reward. This is execution feedback—replayable, environment-grounded—not tool-augmented retrieval or detection.

3. **Visual Verification Loop**: The Solver reasons over images generated by the Coder. The Solver's answers can be verified against the Proposer's intended concepts—creating a closed loop where execution and visual verification drive self-improvement.

4. **GRPO Training with Execution Outcomes**: All three roles are trained with GRPO (Group Relative Policy Optimization). Rewards incorporate execution feedback (did the code run? did it produce valid images?) and visual verification (does the Solver's answer match the Proposer's ground truth?). This is execution-grounded RL.

5. **Contrast with Q2 (Tool-Augmented)**: MM-Zero does not merely call external tools for retrieval or detection. The Coder's code *is* the reasoning trace—it is executed to produce training data. The execution is central to the framework, not auxiliary.

6. **Contrast with Q1 (Text-Only)**: The Coder's output is not free-form text; it is executable code. The framework depends on code execution to generate images and to provide reward signals. Without execution, MM-Zero cannot function.

## Key Contributions

1. **First Zero-Data Self-Evolving VLM Framework**: MM-Zero enables VLMs to evolve from zero human-annotated data. The Proposer-Coder-Solver pipeline generates synthetic concepts, renders them via code execution, and trains on self-generated multimodal data—a fully autonomous data generation and training loop.

2. **Three-Role Architecture with Execution Feedback**: Proposer generates concepts and questions; Coder converts concepts to executable Python/SVG code and renders images; Solver performs multimodal reasoning. Execution feedback (code runs, images are valid) and visual verification (Solver answers match Proposer's intent) are integrated into GRPO rewards.

3. **GRPO for All Roles**: All three roles are trained with GRPO, enabling coordinated self-improvement. The framework demonstrates that execution feedback and visual verification can drive VLM evolution without human annotation.

## Verifiability Analysis

### Grounding Strength
**Assessment:** Very High

The Coder's code produces concrete images—execution grounds the representation in actual visual output. The Solver's reasoning is grounded in these generated images. The Proposer's concepts are verified by whether the Coder can render them and the Solver can answer correctly. Execution provides unambiguous grounding.

### Checkability
**Assessment:** Very High

- **Code**: Python/SVG syntax is checkable by interpreter; execution success/failure is deterministic.
- **Images**: Generated images can be inspected; visual quality and concept alignment can be evaluated.
- **Solver answers**: Can be compared against Proposer's intended answers (self-consistency check).
- **Execution traces**: Code + execution logs provide full audit trail.

### Replayability
**Assessment:** Very High

Given the Coder's code and execution environment, image generation is fully reproducible. The entire pipeline (Proposer → Coder → Solver) can be re-run with fixed seeds. Execution is deterministic for given code and environment.

### Faithfulness Risk
**Assessment:** Low

- **Coder**: Code either executes or fails; generated images are ground truth (not hallucinated).
- **Solver**: Reasons over actual generated images; no "explain without seeing" for those images.
- **Proposer**: Concepts are verified by downstream Coder and Solver performance—closed-loop verification.

### Robustness
**Assessment:** Moderate

- **Strengths**: Self-evolving loop can discover and correct failure modes; execution feedback provides concrete signals.
- **Weaknesses**: Code execution may fail (syntax errors, runtime errors, unsupported operations); generated images may not match real-world distribution; bootstrap from zero data may lead to distribution drift.

### Cost/Latency
**Assessment:** High

- **Code execution**: Each Coder output requires execution (Python interpreter, SVG renderer).
- **Image generation**: Rendering may be slow for complex scenes.
- **GRPO training**: Multiple rollouts per sample; execution + Solver forward passes multiply cost.
- **Zero data**: No human annotation cost, but compute cost is significant.

### Security
**Assessment:** High Risk (Requires Mitigation)

- **Code execution**: Running model-generated Python/SVG poses security risks—arbitrary code execution, file system access, network calls.
- **Mitigation**: Sandboxed execution (restricted interpreter, no file/network access), whitelisted libraries, timeout limits.
- **Prompt injection**: Proposer/Solver prompts could be manipulated; Coder could generate malicious code if not constrained.

---

## Failure Modes

1. **Code Execution Failures**: The Coder may generate syntactically or semantically invalid code—wrong API usage, unsupported operations, infinite loops. Execution failures provide negative feedback but may slow or destabilize training.

2. **Concept-Image Mismatch**: The Coder may render images that do not match the Proposer's intended concepts. The Solver may then learn from misaligned data, propagating errors through the self-evolving loop.

3. **Distribution Drift**: With zero human data, the model may drift toward concepts and visual styles that are easy to generate and solve, but diverge from real-world distribution. Long-term self-evolution may amplify this drift.

4. **Proposer Quality Bottleneck**: If the Proposer generates trivial or repetitive concepts, the entire pipeline is limited. The Proposer's diversity and difficulty directly constrain Coder and Solver improvement.

5. **SVG/Python Rendering Limitations**: Python/SVG may not support all visual concepts (e.g., photorealistic scenes, complex 3D). The framework may be biased toward renderable, diagram-like content.

6. **GRPO Coordination Failures**: The three roles may not coordinate well—e.g., Coder optimizes for easy-to-render concepts while Solver needs harder examples. Multi-agent credit assignment can be challenging.

## Notes

The three-role design (Proposer, Coder, Solver) provides a clean separation of concerns. Security of code execution is critical for deployment.
