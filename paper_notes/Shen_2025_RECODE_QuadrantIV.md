# RECODE: Reasoning Through Code Generation for Visual Question Answering

## Basic Information

**Title**: RECODE: Reasoning Through Code Generation for Visual Question Answering

**Authors**: Junhong Shen, Mu Cai, Bo Hu, Ameet Talwalkar, David A. Ross, Cordelia Schmid, Alireza Fathi

**Venue**: arXiv 2025 (under review at ICLR 2026)

**Year**: 2025

**Link**: https://arxiv.org/abs/2510.13756

---

## Abstract Summary

RECODE proposes "derendering" — reverse-engineering structured visuals (charts, diagrams, geometry) into executable code that reproduces them — as a novel modality for verifiable visual reasoning. Instead of directly perceiving image content (which is ambiguous and unverifiable), RECODE generates candidate derendering programs, executes them to render reconstructed images, uses a critic to select the most faithful reconstruction, and iteratively refines the code. The final high-fidelity code representation enables precise calculations and logical inference on symbolic representations rather than raw pixels.

## Methodology Analysis

### Representation Type
- [x] Structured Trace (executable derendering programs: Python code that produces the input image; e.g., matplotlib code for charts, geometric construction code for diagrams)

### Verification Channel
- [x] Tool-Augmented (Python execution environment; image rendering engine)
- [x] Execution Feedback (rendered image vs. original image as visual verification signal; critic selects best derendering by execution output comparison)

### 2×2 Matrix Placement
**Quadrant**: IV

**Justification**:
- **Structured**: The intermediate representation is executable code (Python/matplotlib or geometry DSL) that symbolically encodes the full content of the input image. This is a *stronger* form of structured representation than bounding boxes or program plans — it is a formal specification of the image content itself.
- **Tool/Execution**: Derendering programs are executed (rendered) and the rendering output is compared to the original image. This is external verification: execution confirms whether the symbolic representation faithfully captures the visual content. The critic uses this execution output to select the best program.
- **Replayability**: Extremely high — the derendering program IS a formal specification of the image. Anyone can re-run the program to verify it reproduces the image, and any Q&A on that image can be re-derived symbolically from the code.

---

## Key Contributions

1. Introduces "derendering as a modality" — converting ambiguous pixel-level visual perception into verifiable symbolic reconstruction, enabling the same formal tools used for code verification to apply to visual reasoning
2. Agentic framework with critic-guided iterative refinement: generate → execute → compare → refine → answer
3. Significant gains on CharXiv, ChartQA, and Geometry3K over both non-code methods and methods using code only for auxiliary drawing (highlighting that the key is grounding visual content in symbolic programs, not just using code for annotation)

## Verifiability Analysis

### Grounding Strength
Extremely strong — the derendering program is a pixel-accurate symbolic description of the image. Visual content is grounded in executable code elements (e.g., specific bar heights in matplotlib as float values, geometric angle specifications in degrees). No other Q4 approach achieves this level of grounding completeness.

### Checkability
Extremely high — the criterion for intermediate representation correctness is objective: execute the derendering code, compare rendered output to original image (can be measured by pixel similarity, SSIM, etc.). Unlike other Q4 approaches, the verification signal does not require an LLM.

### Replayability
Extremely high — the derendering program is a complete formal specification. Running it reproduces the image exactly. Any downstream question-answering can be re-derived from the program (e.g., re-reading bar heights from code rather than from image pixels).

### Faithfulness Risk
Near-zero — the derendering program is explicitly verified against the image by execution. Unlike text CoT, unfaithful representation is detectable: if the rendered program doesn't match the image, the critic rejects it. The iterative refinement loop actively eliminates unfaithful representations.

### Robustness
High for structured visuals (charts, geometry) with well-defined grammar; low for natural images or non-standard visual formats where derendering has no clear grammar. Performance depends on whether the image was generated by a known class of programs.

### Cost/Latency
High — the generate-execute-critique-refine loop requires multiple forward passes plus image rendering and comparison at each iteration. Most expensive among Q4 approaches in terms of per-query latency.

### Security
Medium — executing Python code in a rendering environment. Adversarial images could be constructed to trigger code generation attacks (though derendering programs are constrained to visualization libraries, limiting attack surface).

---

## Failure Modes

1. **Derendering failure**: For highly stylized, non-standard, or hand-drawn visuals, no faithful derendering program may exist in the assumed code grammar (e.g., artistic bar charts with non-standard colors). The framework degrades to standard visual perception when derendering fails.
2. **Color/encoding errors**: If the derendering program gets color encodings wrong (e.g., misidentifying which bar color maps to which data series), all downstream numerical reasoning is based on an incorrect but visually similar chart representation — error is silent (rendering looks similar to original).
3. **Cost blowup with iterative refinement**: For complex charts with many data series, iterative refinement may require many cycles. No clear termination criterion guarantees convergence.

## Evaluation

### Metrics Used
- [x] Answer Accuracy
- [x] Trace Replayability (implicit — derendering programs are replayable by construction)
- [x] Evidence Attribution (derendering code attributes every value to a specific program element)

### Benchmarks
- CharXiv (chart understanding and reasoning)
- ChartQA (chart question answering)
- Geometry3K (geometric reasoning)

### Key Results
- RECODE significantly outperforms: (1) methods not using code, (2) methods using code only for drawing auxiliary lines or cropping
- Key finding: the critical factor is *grounding visual content in executable code*, not just using code as annotation tool

## Quotes & Key Insights

> "Pixel-based perception lacks a mechanism for verification." — Core problem statement that motivates the entire approach.

> "Grounding visual perception in executable code provides a new path toward more accurate and verifiable multimodal reasoning."

Key insight: **derendering as a route to verifiability** — if you can write a program that generates the image, you have a machine-checkable specification of its content. This is the most rigorous form of Q4 structured representation because the ground truth for the representation is objectively verifiable (does the code reproduce the image?).

## Survey Placement

### Section Placement
- [x] Section 4.4 (Quadrant IV — "derendering as structured trace" design pattern)
- [x] Section 8 (Challenges — limited to structured visual domains; natural images remain hard)

### Narrative Role
RECODE represents the **"formal specification" extreme of Q4**: where the structured representation is so complete it can regenerate the input image. It makes the strongest verifiability argument in Q4 but has the most constrained domain applicability.

### Comparison Points
- Excels at: replayability, checkability, faithfulness (objective verification via rendering)
- Weaker at: generalization to unstructured images; cost/latency; domains without known rendering grammars
